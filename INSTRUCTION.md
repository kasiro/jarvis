# 📜 INSTRUCTION.md - Инструкция по работе с документацией

## 🎯 Обязательное правило

**ПРИ КАЖДОМ ВОПРОСЕ/ЗАДАЧЕ:**

1. **Проверить `QWEN.md`** - общая информация о проекте
2. **Проверить `EVERY_CHECK.md`** - актуальный чек-лист
3. **Проверить `ADD.md`** - как добавлять новое
4. **Проверить `FIX.md`** - известные проблемы

---

## 🔄 Алгоритм работы

### 1. Получение задачи

```
User: <вопрос или задача>
```

### 2. Проверка документации

**Обязательно проверить:**

| Файл | Зачем | Когда |
|------|-------|-------|
| `QWEN.md` | Общая информация, архитектура | **Всегда** |
| `EVERY_CHECK.md` | Чек-лист, проверки | **Всегда** |
| `TODO_LIST.md` | 📝 Напоминание о задачах пользователя | **При запуске** |
| `ADD.md` | 📝 Список функций для реализации | При планировании работы |
| `FIX.md` | 🔧 Список проблем для исправления | При планировании работы |
| `resources/commands/ADD.md` | 📝 Список команд для реализации | При планировании работы |
| `resources/commands/FIX.md` | 🔧 Список проблем с командами | При планировании работы |

### 3. Выполнение задачи

1. **Прочитать документацию**
2. **Сверить с текущим состоянием**
3. **Выполнить задачу**
4. **Обновить документацию** (если нужно)

### 4. Обновление TODO списков

**После ЛЮБЫХ изменений:**

```markdown
# В соответствующий TODO файл:

## ADD.md
- Если реализовал функцию → отметить выполненной ✅

## FIX.md
- Если исправил проблему → отметить выполненной ✅

## EVERY_CHECK.md
- Если изменилась проверка → обновить чек-лист

## QWEN.md
- Если изменилась архитектура → обновить историю
```

---

## 📋 Примеры

### Пример 1: Пользователь говорит "Нужно добавить новую функцию"

**Действия:**
1. ✅ Проверить `ADD.md` - есть ли уже такая задача
2. ✅ Проверить `QWEN.md` - архитектура проекта
3. ✅ Выполнить задачу
4. ✅ Отметить в `ADD.md` как выполненную ✅

### Пример 2: Пользователь сообщает об ошибке

**Действия:**
1. ✅ Проверить `FIX.md` - есть ли уже такая проблема
2. ✅ Проверить `EVERY_CHECK.md` - чек-лист диагностики
3. ✅ Исправить ошибку
4. ✅ Отметить в `FIX.md` как исправленную ✅

### Пример 3: Пользователь просит исправить команды

**Действия:**
1. ✅ Проверить `resources/commands/FIX.md` - есть ли проблема в списке
2. ✅ Проверить `EVERY_CHECK.md` - чек-лист проверки команд
3. ✅ Исправить команды
4. ✅ Отметить в `resources/commands/FIX.md` как исправленную ✅

---

## 🎯 Золотые правила

1. **НИКОГДА не отвечай без проверки документации**
2. **ВСЕГДА обновляй TODO списки** после выполнения задачи
3. **ВСЕГДА сверяйся с `QWEN.md` при запуске**
4. **ВСЕГДА выполняй `EVERY_CHECK.md` перед работой**
5. **ВСЕГДА проверяй `TODO_LIST.md`** - напоминай о незавершённых задачах!
6. **НЕ РЕДАКТИРУЙ `TODO_LIST.md`** - этот файл меняет только пользователь
